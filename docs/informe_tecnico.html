1. Introducción y Objetivos

Este proyecto implementa una plataforma completa de análisis de reseñas de restaurantes en Panama, integrando técnicas de Machine Learning, procesamiento de lenguaje natural con LLMs, y visualización de datos mediante dashboards interactivos. La plataforma extrae reseñas reales de Google Maps (via Playwright) y Yelp Fusion API, las procesa a través de un pipeline ETL, aplica modelos de clustering y regresión, y presenta los resultados en un dashboard Streamlit y archivos Power BI.

Objetivos del Proyecto

  1. Recolección de datos: Obtener más de 1000 reseñas reales de restaurantes panameños desde múltiples fuentes legítimas.
  2. Análisis de sentimiento: Extraer sentimiento por aspecto (comida, servicio, precio) usando LLMs con respaldo keyword-based.
  3. Machine Learning: Implementar clustering no supervisado (KMeans) y regresión supervisada (Random Forest).
  4. Visualización: Desarrollar dashboards interactivos en Streamlit y Power BI con esquema estrella.
  5. LLM Integration: Incorporar consultas en lenguaje natural y resúmenes automáticos via chat.

3. Arquitectura del Pipeline

El pipeline ETL está orquestado por run_pipeline.py que ejecuta 6 pasos secuenciales:

Google Maps (Playwright) ----+ +---> descargar_datos.py -> limpiar_datos.py Yelp Fusion API -------------+ | analizar_sentimiento.py (LLM + keyword backup) | clustering.py (KMeans) | regresion.py (RandomForest) | exportar_powerbi.py (Estrella) | +----------+----------+ | | Streamlit app.py Power BI (.pbix) (+ chat LLM) (+ KPIs visuales)

3.1 Paso 1: Descarga de Datos (descargar_datos.py)

Orquesta la obtención de datos desde múltiples fuentes en orden de prioridad:

  1. Playwright: Ejecuta scraper en Google Maps para 20 restaurantes de Panama (~120 reseñas c/u).
  2. Selenium: Fallback automático si Playwright falla. Obtiene nombres reales + reseñas generadas.
  3. Yelp Fusion API: Obtiene reseñas legales vía API oficial (gratuita, 500 llamadas/día).
  4. Dataset de respaldo: CSV precargado con reseñas en español.

La salida es data/reviews.csv con columnas: restaurante, categoria, barrio, rating, reseña, sentimiento, fecha.

3.2 Paso 2: Limpieza (limpiar_datos.py)

Transformaciones aplicadas:

  • Limpieza de texto: eliminación de caracteres especiales, normalización de espacios.
  • Filtrado: reseñas sin texto válido (menos de 5 caracteres) o sin rating son descartadas.
  • Deduplicación: basada en (restaurante + primeros 50 caracteres de reseña).
  • Clasificación por rating: bueno (>3.5), neutral (2.0-3.5), malo (<=2.0).
  • Salida: data/reviews_clean.csv con reseña_limpia, longitud_reseña, categoria_rating.

3.3 Paso 3: Análisis de Sentimiento (analizar_sentimiento.py)

Extrae sentimiento para 3 aspectos (comida, servicio, precio) usando dos modalidades:

  • LLM (recomendado): Usa Claude 3 Haiku o GPT-4o-mini para extraer sentimiento con alta precisión semántica. Límite de 50 llamadas por ejecución para control de costos. El prompt pide JSON estructurado con sentimiento_general y sentimiento por aspecto más palabras clave.
  • Keyword (fallback): Léxico con 50+ términos por aspecto en español. Cuenta ocurrencias de palabras positivas/negativas para determinar sentimiento. Siempre se usa para reseñas más allá del límite LLM.

Salida: data/reviews_con_sentimiento.csv con columnas de sentimiento para cada aspecto y palabras clave extraídas.

3.4 Paso 4: Clustering (clustering.py)

Agrupa restaurantes por desempeño usando KMeans. Features normalizadas con StandardScaler: rating_promedio (peso 0.4), num_resenas (popularidad, peso 0.2), pct_positivo (peso 0.6). Los clusters se etiquetan ordenando centroides por calidad: Excelente, Bueno, Regular, Malo.

3.5 Paso 5: Regresión (regresion.py)

Random Forest Regressor para predecir rating numérico. Features: longitud del texto, sentimiento general, sentimiento por aspecto (comida/servicio/precio), restaurante (codificado). Train/test 80/20. Evalúa con MAE, RMSE, R². Guarda importancia de features y tabla de predicciones.

3.6 Paso 6: Exportación Power BI (exportar_powerbi.py)

Genera 5 CSVs en esquema estrella (FactResenas, DimRestaurante, DimFecha, DimSentimiento, DimRating) y llama al compilado tools/BuildPBIX.exe (C# con librerías TOM de Power BI Desktop) para crear data/RestaurantReviews.pbix con el modelo tabular completo: tablas, columnas, relaciones y 14 medidas DAX.

Nota: El .pbix solo se genera la primera vez. En ejecuciones posteriores, se conserva para mantener los visuales del dashboard. Solo se actualizan los CSVs; el usuario abre Power BI y hace clic en "Actualizar".

4. Origen y Tratamiento de los Datos

4.1 Google Maps (Playwright)

El scraper scraper_playwright.py automatiza un navegador Chromium headless para:

  1. Buscar "restaurantes en Panama" en Google Maps.
  2. Extraer URLs de restaurantes del feed de resultados (scroll para carga completa).
  3. Navegar a cada restaurante y hacer clic en la pestaña "Opiniones".
  4. Hacer scroll para cargar reseñas mediante lazy-loading (hasta 120 reseñas por restaurante).
  5. Extraer para cada reseña: texto, rating (estrellas), reseñador y fecha.

Configuración: locale es-PA, user-agent de escritorio, viewport 1920x1080.

4.2 Yelp Fusion API

scraper_yelp.py consume la API oficial de Yelp (gratuita con registro en developers.yelp.com):

  1. GET /businesses/search: busca restaurantes en Panama con paginación (hasta 50 negocios).
  2. GET /businesses/{id}/reviews: obtiene hasta 3 reseñas por negocio.
  3. Combina con datos de Google Maps para un dataset más completo.

4.3 Dataset Actual

El dataset final contiene reseñas reales y auténticas de restaurantes panameños. No se utilizan reseñas generadas sintéticamente. Las reseñas pasan por un proceso de limpieza y deduplicación antes del análisis.

5. Modelos de Machine Learning

5.1 KMeans Clustering (No Supervisado)

Descripción

KMeans agrupa restaurantes en 4 clusters basados en 3 características numéricas normalizadas con StandardScaler: rating promedio, porcentaje de reseñas positivas, y número de reseñas (popularidad). Los clusters se etiquetan automáticamente según la calidad del centroide.

Hiperparámetros

ParámetroValorJustificación
n_clusters4Número manejable de categorías de desempeño. Suficiente para diferenciar sin sobre-segmentar con solo 5-20 restaurantes.
random_state42Reproducibilidad de resultados.
n_init10Evita óptimos locales con múltiples inicializaciones.

Justificación de la Elección

KMeans fue seleccionado porque:

  1. Interpretabilidad: Los clusters resultantes tienen significado directo (Excelente, Bueno, Regular, Malo), facilitando la comunicación con stakeholders.
  2. Escalabilidad: Computacionalmente eficiente para el volumen actual de datos (~5-20 puntos).
  3. Simplicidad: Pocos hiperparámetros que ajustar, resultado determinista con random_state fijo.
  4. Alternativas descartadas: DBSCAN requiere densidad variable que no tenemos con pocos restaurantes; Clustering Jerárquico es overkill para este volumen y menos intuitivo para etiquetar.

5.2 Random Forest Regressor (Supervisado)

Descripción

Random Forest Regressor predice el rating numérico (1-5) de una reseña individual basado en 6 features: longitud del texto, sentimiento general (0/1/2), sentimiento por aspecto (comida, servicio, precio), y restaurante (codificado con LabelEncoder).

Hiperparámetros

ParámetroValorJustificación
n_estimators100Balance entre precisión y tiempo de entrenamiento. Suficiente para estabilizar el error out-of-bag.
random_state42Reproducibilidad.
n_jobs-1Usa todos los núcleos CPU para entrenamiento paralelo.
train/test80/20Split estándar para datasets pequeños.

Métricas Obtenidas

MétricaValorInterpretación
MAE0.52El error promedio es de ~0.5 estrellas, aceptable para predicción de ratings.
RMSE0.83Penaliza errores grandes; el RMSE < 1 indica que la mayoría de predicciones están dentro de 1 estrella.
R²0.26El modelo explica el 26% de la varianza. Moderado, esperado dado que features textuales simples tienen capacidad predictiva limitada.

Importancia de Features

FeatureImportanciaInterpretación
longitud0.58La longitud del texto es el predictor más fuerte: reseñas largas tienden a tener ratings más extremos.
sentimiento_num0.16El sentimiento general captura la polaridad de la reseña.
restaurante_id0.14El restaurante específico influye: algunos tienen ratings consistentemente más altos.
sentimiento_comida_num0.04El sentimiento sobre comida tiene impacto moderado.
sentimiento_servicio_num0.04Similar al de comida.
sentimiento_precio_num0.04El precio es el aspecto menos predictivo del rating.

Justificación de la Elección

Random Forest Regressor fue seleccionado porque:

  1. Features mixtos: Maneja naturalmente variables numéricas (longitud) y categóricas (restaurante codificado, sentimientos ordinales).
  2. Robustez a outliers: Las reseñas muy positivas o negativas no sesgan el modelo gracias al promediado de árboles.
  3. No requiere escalado: A diferencia de SVM o regresión lineal, Random Forest es invariante a la escala de features.
  4. Feature importance: Provee interpretabilidad directa de qué factores influyen más en el rating.
  5. Alternativas descartadas: Regresión Lineal asume linealidad que no tenemos. SVM es más costoso y menos interpretable. Gradient Boosting sobreajusta con datos pequeños.

7.6 Cómo Funcionan las Medidas DAX

DIVIDE vs División Directa

Todas las medidas de porcentaje usan DIVIDE() en lugar de /. Esto es porque DIVIDE() retorna 0 cuando el denominador es 0 (evita errores #DIV/0!), mientras que / generaría un error que rompería el visual.

CALCULATE y Filtros

Las medidas como % Positivo Comida usan CALCULATE() para modificar el contexto de filtro. Por ejemplo:

% Positivo Comida = DIVIDE(
    CALCULATE(COUNTROWS(FactResenas), DimSentimiento[sentimiento_comida] = "positivo"),
    [Total Reseñas]
)

CALCULATE cuenta solo las reseñas donde sentimiento_comida = "positivo", ignorando el resto. Luego se divide entre el total para obtener el porcentaje.

Scoring Compuesto — Desglose

Scoring Compuesto =
    [Rating Promedio] * 0.5                    -- Calidad (50% del peso)
    + [% Positivo] * 0.3                       -- Sentimiento (30% del peso)
    + DIVIDE(COUNTROWS(FactResenas),           -- Actividad reciente (20%)
             MAX(FactResenas[id_resena])) * 0.2 * 5

El tercer término mide la proporción de reseñas recientes vs el ID más alto, escalar a 0-5. Esto recompensa restaurantes con actividad reciente.

7.8 Actualización de Datos

El pipeline está diseñado para ejecutarse múltiples veces sin destruir el trabajo visual:

  1. La primera vez: se ejecuta exportar_powerbi.py que genera los CSVs y el archivo .pbix con el modelo tabular.
  2. Se abre el .pbix en Power BI Desktop y se crean todos los visuales (4 páginas).
  3. En ejecuciones posteriores del pipeline: solo se actualizan los CSVs en data/powerbi/.
  4. En Power BI Desktop: se hace clic en Refresh (flecha circular en la cinta Home).
  5. Los datos se recargan y todos los visuales se conservan intactos — no es necesario recrear nada.

11. Preguntas Frecuentes del Examen (Pregunta → Respuesta)

General del Proyecto

P: ¿Qué hace este proyecto exactamente?

R: Es una plataforma completa de análisis de reseñas de restaurantes en Panamá. Extrae reseñas reales de Google Maps (usando Playwright para scraping) y Yelp Fusion API (usando requests), las procesa con un pipeline ETL de 6 pasos, aplica Machine Learning (clustering con KMeans y regresión con Random Forest), analiza sentimiento por aspecto (comida, servicio, precio) usando LLMs como Claude o GPT, y presenta todo en dos dashboards: uno interactivo con Streamlit y otro profesional con Power BI usando esquema estrella.

P: ¿Cuántas reseñas se recolectaron y de dónde vienen?

R: Se recolectaron más de 1,264 reseñas reales de 10 restaurantes en Panamá. Las fuentes son Google Maps (principal, vía scraping con Playwright) y Yelp Fusion API (complementaria, vía requests). No se usaron reseñas generadas sintéticamente; todas son auténticas de usuarios reales.

P: ¿Por qué usaron Python y no otro lenguaje?

R: Python fue elegido por su ecosistema robusto de librerías para cada componente: pandas/numpy para datos, scikit-learn para ML, playwright/selenium para scraping, streamlit para dashboards, y múltiples clientes para LLMs (anthropic, openai, google-generativeai, groq). Ningún otro lenguaje tiene esta concentración de librerías especializadas para ciencia de datos y NLP.

P: ¿Qué es un pipeline ETL y cómo funciona el de este proyecto?

R: ETL significa Extract, Transform, Load (Extraer, Transformar, Cargar). Es un proceso que toma datos de múltiples fuentes, los limpia y transforma, y los carga en un formato listo para análisis. Nuestro pipeline tiene 6 pasos secuenciales orquestados por run_pipeline.py:

1. Descargar (extract): scraper_playwright.py obtiene datos de Google Maps, scraper_yelp.py de Yelp API.
2. Limpiar (transform): limpiar_datos.py elimina reseñas vacías, deduplica, normaliza texto.
3. Sentimiento (transform): analizar_sentimiento.py usa LLMs + keywords para clasificar sentimiento por aspecto.
4. Clustering (transform): clustering.py agrupa restaurantes con KMeans.
5. Regresión (transform): regresion.py predice ratings con Random Forest.
6. Exportar (load): exportar_powerbi.py genera CSVs en esquema estrella para Power BI.

Scraping y Datos

P: ¿Qué es Playwright y por qué lo usan para scraping?

R: Playwright es una librería de Microsoft para automatización de navegadores web. La usamos porque Google Maps carga contenido dinámicamente con JavaScript (lazy-loading en la pestaña de opiniones). Playwright puede: (1) abrir un navegador Chromium headless (sin ventana visible), (2) navegar a la pestaña "Opiniones" de cada restaurante, (3) hacer scroll automático para que carguen más reseñas, (4) extraer el texto, rating, fecha y nombre del reseñador de cada opinión. Opera con locale es-PA, viewport 1920x1080, y user-agent de escritorio para simular un usuario real.

P: ¿Qué es Selenium y por qué lo tienen como fallback?

R: Selenium es otra librería de automatización de navegadores, más antigua que Playwright. Lo usamos como respaldo por si Playwright falla (por ejemplo, si Chromium no se puede instalar). Selenium usa ChromeDriver Manager para descargar automáticamente el driver compatible con el Chrome del usuario. Si Playwright funciona, Selenium no se ejecuta.

P: ¿Qué es la API de Yelp y cómo la consumen?

R: Yelp Fusion API es una API REST gratuita (con registro en developers.yelp.com) que permite buscar negocios y obtener reseñas. Nuestro script scraper_yelp.py hace dos tipos de llamadas: (1) GET /businesses/search para buscar restaurantes en Panamá con paginación (hasta 50 negocios), y (2) GET /businesses/{id}/reviews para obtener hasta 3 reseñas por negocio. Usa la librería requests de Python para hacer las llamadas HTTP con headers de autenticación.

P: ¿Qué limpieza se le hace a los datos?

R: En limpiar_datos.py se aplican 4 transformaciones: (1) Limpieza de texto: eliminación de caracteres especiales y normalización de espacios múltiples. (2) Filtrado: reseñas con menos de 5 caracteres o sin rating son descartadas. (3) Deduplicación: basada en la combinación de restaurante + los primeros 50 caracteres de la reseña. (4) Clasificación por rating: bueno (>3.5), neutral (2.0-3.5), malo (≤2.0). La salida es data/reviews_clean.csv.

Machine Learning

P: ¿Qué es KMeans y por qué lo usaron?

R: KMeans es un algoritmo de clustering no supervisado que agrupa datos en K grupos basándose en la distancia euclidiana entre puntos. Funciona así: (1) Selecciona K centroides aleatoriamente. (2) Asigna cada punto al centroide más cercano. (3) Recalcula los centroides como el promedio de los puntos asignados. (4) Repite hasta que los centroides no cambien. Lo usamos porque: es interpretable (los clusters tienen significado directo: Excelente, Bueno, Regular, Malo), es eficiente para pocos datos (10 restaurantes), y tiene pocos hiperparámetros. Usamos n_clusters=4, random_state=42 para reproducibilidad, y n_init=10 para evitar óptimos locales.

P: ¿Qué features usan para el clustering y por qué esos pesos?

R: Usamos 3 features normalizadas con StandardScaler: (1) rating_promedio (peso 0.4): representa la calidad percibida. (2) num_resenas (peso 0.2): representa la popularidad. (3) pct_positivo (peso 0.6): representa el sentimiento general. El peso más alto en pct_positivo (0.6) se debe a que el sentimiento es el indicador más confiable de satisfacción del cliente, mientras que el rating puede estar sesgado por valores extremos.

P: ¿Qué es Random Forest y cómo predice ratings?

R: Random Forest es un algoritmo de ML supervisado que usa múltiples árboles de decisión para hacer predicciones. Cada árboles se entrena con una muestra aleatoria de los datos (bootstrap) y usa un subconjunto aleatorio de features. La predicción final es el promedio de todos los árboles (para regresión). En nuestro caso, predice el rating numérico (1-5) de una reseña individual basado en 6 features: longitud del texto, sentimiento general, sentimiento por aspecto (comida, servicio, precio), y restaurante codificado. Usamos 100 árboles (n_estimators=100), train/test 80/20, y evaluamos con MAE, RMSE y R².

P: ¿Qué significan las métricas MAE, RMSE y R²?

R: MAE (Mean Absolute Error): error promedio absoluto entre predicciones y valores reales. Nuestro MAE de 0.52 significa que el modelo se equivoca en promedio medio punto de estrella. RMSE (Root Mean Squared Error): similar al MAE pero penaliza más los errores grandes (porque eleva al cuadrado). Nuestro RMSE de 0.83 indica que la mayoría de predicciones están dentro de 1 estrella. R² (Coeficiente de Determinación): indica qué porcentaje de la varianza explica el modelo. Nuestro R² de 0.26 significa que el modelo explica el 26% de la variabilidad del rating. Es moderado, esperado porque features textuales simples tienen capacidad predictiva limitada.

P: ¿Por qué Random Forest y no Regresión Lineal?

R: Random Forest fue elegido porque: (1) Maneja features mixtos (numéricos como longitud y categóricos como restaurante codificado) sin necesidad de escalado. (2) Es robusto a outliers (reseñas muy extremas no sesgan el modelo). (3) Provee feature importance (sabemos qué factores influyen más). (4) No asume linealidad. Regresión Lineal asume relación lineal entre features y target, que no existe en texto. SVM es más costoso computacionalmente y menos interpretable. Gradient Boosting sobreajusta con pocos datos.

P: ¿Qué es StandardScaler y por qué lo usan?

R: StandardScaler es una técnica de preprocesamiento que normaliza las features para que tengan media 0 y desviación estándar 1. Se calcula como: (x - media) / desviación_estándar. Lo usamos en KMeans porque el algoritmo se basa en distancias euclidianas. Sin escalado, num_resenas (valores entre 15-150) dominaría sobre rating_promedio (valores entre 4-5) simplemente por tener valores más grandes. Con StandardScaler, todas las features contribuyen equitativamente al cálculo de distancias.

Análisis de Sentimiento y LLMs

P: ¿Qué es un LLM y cómo lo usan para análisis de sentimiento?

R: LLM significa Large Language Model (Modelo de Lenguaje Grande). Son modelos de IA entrenados con billions de texto que pueden entender y generar lenguaje humano. Los usamos para analizar sentimiento porque entienden el contexto y las sutilezas del lenguaje (ej: "la comida estaba bien pero el servicio fue un desastre" tiene sentimiento mixto). El flujo es: (1) Enviamos la reseña con un prompt que pide JSON estructurado. (2) El LLM responde con sentimiento_general, sentimiento_comida, sentimiento_servicio, sentimiento_precio y palabras_clave. (3) Parseamos la respuesta JSON y la guardamos.

P: ¿Qué es el fallback basado en keywords?

R: Es un método de respaldo que no usa IA. Tenemos un léxico de 50+ términos en español por cada aspecto (comida, servicio, precio). Por ejemplo, para comida: "delicioso", "rico", "sabroso" son positivos; "malo", "crudo", "frío" son negativos. El algoritmo cuenta cuántas palabras positivas y negativas hay en la reseña. Si hay más positivas → positivo. Si hay más negativas → negativo. Si hay igualdad o ninguna palabra del léxico → neutral. Se usa siempre para reseñas más allá del límite de 50 llamadas LLM, y como verificación cruzada.

P: ¿Qué proveedores de LLM soportan y por qué?

R: Soportamos 4 proveedores en orden de prioridad: (1) Anthropic Claude 3 Haiku (pago, el más preciso para análisis). (2) OpenAI GPT-4o-mini (pago, buena alternativa). (3) Google Gemini 1.5 Flash (gratuito con cuota). (4) Groq Llama 3 70B (gratuito, el más rápido). La función get_available_provider() detecta automáticamente qué API key está configurada y usa el primero disponible. Esto garantiza que el sistema funcione sin depender de un solo servicio.

P: ¿Qué es el rate limiting y la caché LRU?

R: Rate limiting: controla cuántas llamadas a la API se hacen por segundo (1 llamada/segundo por proveedor). Esto evita ser bloqueado por los proveedores de LLM. Caché LRU (Least Recently Used): almacena hasta 200 respuestas de LLM en memoria. Si alguien hace la misma pregunta dos veces, la segunda vez usa la respuesta cacheada en lugar de hacer una nueva llamada a la API. Esto reduce costos y tiempo de respuesta.

Power BI y Esquema Estrella

P: ¿Qué es un esquema estrella?

R: Es un diseño de base de datos dimensional donde hay una tabla central de hechos (FactResenas) rodeada por tablas de dimensiones (DimRestaurante, DimFecha, DimSentimiento, DimRating). La tabla de hechos contiene los datos numéricos y las foreign keys. Las dimensiones contienen los atributos descriptivos. Se llama "estrella" porque la estructura se ve como una estrella con la tabla de hechos en el centro. Es ideal para Power BI porque permite filtros rápidos y agregaciones eficientes.

P: ¿Qué son las measures DAX y cómo funcionan?

R: DAX (Data Analysis Expressions) es el lenguaje de fórmulas de Power BI. Las measures son cálculos virtuales que se calculan en tiempo real cuando interactúas con el dashboard. No existen en el CSV, se calculan sobre la marcha. Ejemplos: Total Reseñas = COUNTROWS(FactResenas) cuenta todas las filas. % Positivo = DIVIDE(CALCULATE(COUNTROWS(FactResenas), DimSentimiento[sentimiento_general] = "positivo"), [Total Reseñas]) calcula el porcentaje de reseñas positivas. La función DIVIDE evita errores de división por cero.

P: ¿Qué hace el gauge de Scoring Compuesto?

R: El gauge (velocímetro) muestra un índice compuesto de 0 a 5 que combina tres factores: (1) Rating Promedio × 0.5 (50% de peso): la calidad percibida. (2) % Positivo × 0.3 (30% de peso): el sentimiento general. (3) Actividad reciente × 0.2 (20% de peso): la proporción de reseñas recientes vs totales. La fórmula completa es: [Rating Promedio] * 0.5 + [% Positivo] * 0.3 + DIVIDE(COUNTROWS(FactResenas), MAX(FactResenas[id_resena])) * 0.2 * 5. El tercer término mide qué tan activo está el restaurante recientemente.

P: ¿Cómo funciona el heatmap de satisfacción?

R: Es una matrix de Power BI donde las filas son restaurantes y las columnas son los 3 aspectos (comida, servicio, precio). Cada celda muestra el porcentaje de sentimiento positivo con formato condicional: verde intenso = alto (cerca de 100%), blanco = medio, rojo = bajo (cerca de 0%). Se configura en Format → Conditional formatting → Background color con esquema Diverging (rojo-blanco-verde). Permite identificar visualmente los puntos débiles de cada restaurante.

P: ¿Por qué el scatter chart no necesita bucket "Values" en mi versión de Power BI?

R: En versiones recientes de Power BI (2026), el scatter chart puede funcionar sin el bucket "Details" o "Values". Si la tabla DimRestaurante ya tiene una fila por restaurante (dat granular), Power BI muestra automáticamente un punto por cada fila. El bucket "Values" en algunas versiones es para measures agregadas, no para detalles. Si cada punto representa un restaurante y los ejes X/Y tienen métricas por restaurante, el gráfico funciona correctamente sin necesidad de un campo adicional.

Streamlit y Dashboard

P: ¿Qué es Streamlit y por qué lo usan?

R: Streamlit es un framework de Python para crear dashboards interactivos con código minimalista. Lo usamos porque: (1) Permite crear interfaces web con scripts Python puros (sin HTML/CSS/JS). (2) Tiene widgets nativos (sliders, selects, tabs). (3) Se integra directamente con pandas, matplotlib y我们的 LLM utils. (4) Soporta chat interactivo (st.chat_message). (5) Es fácil de deployar con streamlit run app.py. Nuestro dashboard tiene 5 pestañas: Resumen General, Análisis por Restaurante, Clustering, Recomendaciones y Asistente LLM.

P: ¿Cómo funciona el chat con LLM en Streamlit?

R: En la pestaña "Asistente LLM", el usuario escribe preguntas en lenguaje natural (ej: "¿Qué restaurante tiene mejor servicio?"). El flujo es: (1) Se captura el mensaje con st.chat_input. (2) Se construye un contexto con estadísticas agregadas de los DataFrames (no los datos crudos). (3) Se envía al LLM disponible (Claude, GPT, Gemini o Groq). (4) El LLM responde basándose exclusivamente en los datos reales. (5) Se muestra la respuesta con st.chat_message. El chat mantiene historial en session_state para conversaciones continuas.

Arquitectura y Diseño

P: ¿Por qué el pipeline es secuencial y no paralelo?

R: El pipeline es secuencial porque cada paso depende del anterior: (1) No puedes limpiar datos que no has descargado. (2) No puedes analizar sentimiento de datos no limpios. (3) No puedes hacer clustering sin sentimientos calculados. (4) No puedes exportar a Power BI sin los clusters y regresión. Sin embargo, dentro de cada paso hay paralelismo: por ejemplo, el scraping de múltiples restaurantes puede hacerse concurrente, y las llamadas LLM se ejecutan en lote.

P: ¿Cómo se mantiene la actualización de datos?

R: El pipeline está diseñado para ejecutarse múltiples veces: (1) La primera vez se genera el .pbix con el modelo tabular completo. (2) En ejecuciones posteriores, solo se actualizan los CSVs en data/powerbi/. (3) En Power BI Desktop se hace clic en Refresh para recargar los datos. (4) Los visuales se conservan intactos — no es necesario recrear nada. Esto permite agregar nuevos restaurantes o reseñas sin perder el trabajo visual.

P: ¿Qué es el PAT (Personal Access Token) de Yelp y cómo se usa?

R: El PAT es un token de autenticación que obtienes al registrarte en developers.yelp.com. Se guarda en el archivo .env como YELP_API_KEY. El script scraper_yelp.py lo lee con python-dotenv y lo envía en el header de cada llamada HTTP: Authorization: Bearer {YELP_API_KEY}. La API es gratuita con un límite de 500 llamadas por día. Sin el token, la API rechaza las peticiones con error 401.

P: ¿Qué diferencia hay entre Streamlit y Power BI en este proyecto?

R: Son complementarios: Streamlit es para interactividad en tiempo real: filtros dinámicos, chat con LLM, gráficos que se actualizan al instante. Es ideal para análisis exploratorio y demostraciones. Power BI es para visualizaciones profesionales y presentaciones ejecutivas: esquema estrella, measures DAX, formato condicional, heatmaps. Es ideal para reportes formales. Ambos usan los mismos datos CSV pero con diferentes niveles de interactividad y presentación.
